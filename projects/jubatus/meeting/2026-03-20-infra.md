## Temi trattati

### 1. Deploy su account AWS del cliente

L'applicazione Jubatus verrà deployata direttamente sull'account AWS del cliente, non su un account LAIF separato. Si utilizzerà la **stessa VPC** già presente, evitando la necessità di VPC peering.

### 2. Stack tecnico LAIF

Presentato lo stack standard di deploy:
- **IaC**: AWS CDK (Python), con script Boto3 per bootstrap iniziale
- **Architettura**: CloudFront → ALB → ECS (autoscaling) + RDS PostgreSQL + S3
- **Database**: RDS PostgreSQL, partenza da istanza T4G micro
- **Template**: tutte le app LAIF partono da template condiviso con gestione utenze, permessi, migrazioni automatiche

### 3. Compatibilità VPC e subnet

Il CDK LAIF deve operare sulla VPC esistente del cliente. Da verificare:
- Compatibilità dello stack CDK con VPC importata (verificato: **supportato**)
- Configurazione subnet (pubbliche con Internet Gateway necessarie per ALB)
- Gestione load balancer (il cliente vorrebbe riutilizzare un ALB esistente)

### 4. Permessi AWS

Approccio concordato: partire con permessi ampi per il setup iniziale, poi restringere progressivamente. Rischio di iterazioni multiple perché AWS può bloccare permessi uno alla volta.

Il cliente creerà:
- Una utenza tecnica per il deploy
- Una utenza personale per accesso diretto

### 5. Integrazioni con database cliente

Necessario accesso ai dati del cliente (ordini, informazioni clienti) per il supporto ticket. Due strategie possibili:
- **Query sincrone** al DB del cliente
- **Job asincroni** ogni 10-15 minuti per estrarre dati

Serve anche accesso cross-account a bucket S3 per recupero immagini e file.

### 6. Gestione email

- **Ricezione**: creare casella email dedicata separata con inoltro automatico per testing
- **Invio**: utilizzo AWS SES con permessi per inviare come supporto@jubatus

### 7. Repository e costi

- Repository inizialmente su GitHub LAIF, possibile trasferimento al cliente in futuro
- Frontend e backend separati (ECR per Docker backend, S3 per sito statico)
- Costo VPC peering (se necessario): sotto 1$/mese

## Verifiche tecniche post-call

### VPC esterna — COMPATIBILE ✅

Il CDK LAIF (`laif_cdk/constructs/networks.py`) supporta già 3 modalità:
- `VpcWithSubnet`: crea VPC nuova (default)
- `VpcImported`: import via `vpc_id` con `Vpc.from_lookup()`
- `VpcFromAttributes`: import con subnet esplicite

Basta passare `vpc_id` al `TemplateStack`.

### Subnet — COMPATIBILE CON CAVEAT ⚠️

Le subnet vengono scoperte automaticamente con il lookup. **Caveat**: l'ALB è hardcoded su subnet PUBLIC (`ecs.py:614`). Le subnet del cliente devono includere subnet pubbliche con Internet Gateway.

### Load Balancer — RICHIEDE MODIFICA CDK ⚠️

Lo stack crea sempre un nuovo ALB. Per riutilizzare l'ALB del cliente serve modificare `laif_cdk/constructs/ecs.py` aggiungendo supporto per ALB importato.

### SES — NON PRESENTE NEL CDK ❌

Non esiste un costrutto SES. Configurazione da fare manualmente o aggiungendo costrutto.
